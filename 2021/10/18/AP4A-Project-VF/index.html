1. Sommaire

La structure principale du projet est implémentée par le diagramme de classes UML sur la dernière page

L’objectif principal du projet est de construire un système de <collection de capteurs-répartiteur recevant puis d’envoi-serveur de réception et de stockage>

1.1 Sensor

Il y a quatre capteurs : température, pression, humidité, lumière

Tous les capteurs sont hérités de la classe des capteurs

En outre, le type de données du capteur de température et de température est un nombre à virgule flottante, le capteur de pression est int, le type de capteur dynamique est bool

Les méthodes nécessaires sont :

-getData() pour obtenir les données
-aleaGenVal() génère des nombres aléatoires

1.2 Planificateur

L’ordonnanceur n’a pas les méthodes nécessaires, mais le serveur et le capteur sont tous les deux une composition relationnelle (composition relationnelle), donc la création du capteur et de l’objet serveur se situe dans l’ordonnanceur

1.3 Serveur

Les principales méthodes du serveur sont :

-Écrire des données dans le journal
-Données de sortie sur l’écran
-Activer/désactiver les données d’écriture et les données de sortie

Chaque capteur différent doit avoir un fichier journal différent

2. Classe de capteur

La méthode aleaGenVal est une méthode protégée et doit être réécrite en sous-classes, donc le mot-clé virtual est ajouté.

3. Classe file de capteur : Température

aleaGenVal est une méthode de substitution, alors ajoutez viatual au début et override deux mots-clés à la fin

Dans chaque sous-type de capteur, le constructeur créera le fichier journal du capteur correspondant, et chaque capteur correspond à un identifiant différent, qui sera utilisé pour écrire les données du capteur correspondant ultérieurement.

Dans la méthode aleaGenval, les données sont traitées/type converties, et les données sont situées dans un intervalle relativement raisonnable pour construire la possibilité d’occurrence dans des situations réelles

4. Class-Scheduler

Tout d’abord, la relation composite doit être reflétée. Lorsque le planificateur est créé, le serveur et les quatre capteurs sont créés ensemble. Lorsque le planificateur est supprimé, le serveur et les capteurs sont également supprimés en conséquence.

La méthode dataTrans_() est utilisée pour transmettre les données de chaque capteur différent au serveur

La méthode trans() consiste à contrôler la fréquence des différentes données entrantes du planificateur

sch_serveur.dataRcv<int>(temperature.getData(),temperature.getID()); Dans l’utilisation de l’instruction de transmission, marquez chaque type différent dans la méthode dataRcv, de sorte que la méthode modèle suivante obtienne la classe correspondante

La méthode trans() utilisait la méthode sleep() auparavant, mais elle bloquera le thread. Chaque méthode ne peut pas envoyer de données au bon moment. Par conséquent, utilisez time_t pour faire la différence de temps entre l’heure actuelle et l’heure de début. 5. Class-Serveur

La méthode changeAct est utilisée pour basculer les données dans le fichier et les afficher à l’écran

Les trois méthodes de dataRcv;consolWrite;fileWrite sont toutes applicables à l’achèvement du modèle, donc la méthode est située dans le fichier d’en-tête

dataRcv détecte si le commutateur d’écriture est activé, puis entre dans la méthode d’écriture correspondante, juge d’abord différents capteurs ou fichiers par identifiant, puis écrit les données

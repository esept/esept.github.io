Rapport de Project AP4B Kanagawa de Informatique En UTBM

[TOC]

Presentation

Kanagawa est un jeu de cartes, les joueur est entre 2 et 4 personne, la carte du jeu a composé par 2 partie, une partie de dessin et une partie de competence, on doit suivi le regle du jeu de prendre les cartes et les utilise comme une carte du jeu ou une carte du competence.

Si on a utilise tous les cartes ou un jouieur a 11 carte de dessin, le jeu va arreter et calcule les points de chaque jouieur par different regle. le jouieur qui a les points le plus haut va gagner ce jeu.

Les travail de Zhaoyang XU :

La partie Model comprende tous les parites de regle du jeu

La partie VIew d’entre du jeu.

Mes partie de diagramme de classes,et la diagramme cas d’utilisation.

Composent

Composent de la carte

Competence :
  • Ajouter un stylo.
  • Ajouter une deplacement du stylo.
  • Prendre une carte.
  • Etre le premier joueur.
Paysage : Les conditions d’ajout d’une carte à dessin
Saison : Augmenter les points
Stylo:

Un stylo peut poser dans une carte de competence, une fois il se poser dans cette carte, le joueur peut ajouter une carte de dessin qui a la meme paysage comme la carte de competence.
Un joueur a minimum 2 stylo et deplacer une stylo vers la gauche ou la droite dans une carte.

Regle du Jeux

Entrer dans le jeu:

Choisir le nombre des joueurs et entrez le nom du joueur.

Le Jeu Commence:

Chaque joueur va prendre une carte de initiation et 2 stylos

Tour du jeu

Tableau

  1. Commerencer de distribuer les cartes, poser les cartes dans premier ligne du tableau
  2. selon l’ordre de joueur, chaqu’un choisir d‘attendre ou une colonne de carte,si il reste un joueur qui n’a pas encore prendre une colonne, la derniere colonne va lui donner automatiquement.
  3. Un joueur peut attendre maximum 2 fois.

Joueur

  1. Le joueur peut utiliser une carte comme une carte de Dessin ou une carte de Compétence, s’il veut l’utilise comme une carte de dessin, le paysage de la partie de dessin de cette carte doit contient dans toutes les paysage de carte de compétence ou les stylos poser.Une carte de dessin est possible d’avoir deux paysages.
  2. S’il une carte qui est utilisé comme une carte de dessin et il y a une compétence, sauf de prendre une carte, les autres vont réaliser automatiquement.
  3. Toutes les cartes doit être utilisée pendant ce tour.

Systeme

Si un joueur a 11 cartes de dessin ou que toutes les cartes sont utilisées, le jeu se termine et calculer les points de chaque joueur.

Diagrammes UML

Diagramme de cas d’utilisation

Le diagramme de cas d’utilisation reflète principalement toutes les opérations du lecteur, et toutes les opérations sont implémentées dans la classe Player :

  1. Sélectionnez une liste de cartes, puis forcez chaque carte à être utilisée comme carte de dessin ou carte de compétence. Lorsque le joueur a la compétence « carte réservée », vous pouvez choisir de conserver la carte.
  2. Ne choisissez pas de carte, attendez le prochain tour de cartes.
  3. Lorsqu’il y a encore des stylos qui ne sont pas placés sur le bureau, vous pouvez mettre les stylos sur le bureau.
  4. Si le joueur peut enregistrer une carte, il peut enregistrer une carte et l’utiliser de manière sélective.
  5. Lorsque le nombre de pinceaux déplacés par le joueur dans ce tour n’est pas nul et qu’au moins un pinceau se trouve sur le bureau, le joueur peut déplacer le pinceau.

Diagramme de classes

Diagramme de Classe de tous les parties de M – Partie Zhaoyang XU

1.Package Card

Ce package est le package de carte,il contient tous les types de cartes.

Lors de la conception de cette partie, j’ai considéré que parce que chaque carte est utilisée séparément, j’ai également divisé la carte en deux parties : la carte de compétence et la carte de dessin. Ensuite, la carte initiale et la carte de dessin du joueur sont composées de cartes de compétence et de cartes de dessin. Cartes ordinaires utilisées en licence.

Une fois que le joueur a obtenu une carte, à l’exception de la carte initiale, le joueur doit choisir une certaine partie à utiliser. Par conséquent, le joueur contiendra deux types de cartes : la carte de dessin et la carte de compétence. L’ordre est également une partie importante de la jeu, j’ai donc utilisé des expressions lambda pour que toutes les cartes de dessin (DCards) et les cartes de compétences (CCards) du joueur héritent de leur ArrayList. Dans ce cas, le calcul des scores et des brosses mobiles Dans les circonstances, l’ordre peut être pleinement respecté.

Lorsque le jeu est initialisé, toutes les cartes (AllCard) utilisent toujours lambda pour hériter de la carte normale (NormalCard). Lorsque la carte est émise, il vous suffit de sélectionner un numéro de carte inférieur au nombre d’objets, puis de le supprimer.

Dans le pack de cartes, chaque classe est très similaire, donc certaines interfaces peuvent être créées.

2.Package View et une partie de package Main

Cette image montre la partie image et la partie arrière-plan lors de l’entrée dans le jeu. La partie la plus importante est de déterminer le nombre de joueurs

Parmi eux, la fonction de multi-threading est utilisée, et EntreGame et MainEntre héritent de la classe Thread.

Dans cette partie, j’utilise DataEntre comme ressource partagée, car le joueur doit sélectionner le nombre de joueurs dans la partie vue, et la partie arrière-plan doit toujours surveiller si le joueur clique sur le bouton “ok”.
Par conséquent, la partie nombreEntre de la partie DataEntre doit être synchronisée. Lorsque le joueur sélectionne un certain nombre, le paramètre nombrePlayer dans EntreGame dans le package View sera modifié, et après avoir cliqué sur ok, ces données seront synchronisées avec DataEntre par l’ensemble method, et MainEntre Ensuite, la méthode get sera toujours utilisée jusqu’à ce que le nombrePlayer change.Après cela, le nombre de joueurs est obtenu, passé au programme principal, exécuté les cartes et entré dans la boucle de jeu.

3.Package Player

Les joueurs sont la catégorie la plus importante dans ce jeu, car le contrôle de l’ensemble du jeu est dirigé par le joueur. Dans la conception, le nom du joueur est automatiquement défini par le système, et tous les joueurs utilisent des nombres pour déterminer leur identité, afin de faciliter les cartes L’utilisation de compétences moyennes (joueur initial).

Parmi eux, le pinceau est un accessoire important pour le joueur. Le système doit ajouter la carte de dessin selon que le décor de l’emplacement du pinceau est le même que la carte de dessin à ajouter. Et le pinceau de chaque joueur est construit dans un ArrayList , via ArrayList.get() Obtenez le pinceau.

La première partie de tous les joueurs est de choisir des cartes. Dans ce processus, chaque joueur peut choisir d’attendre ou de choisir une rangée de cartes. Tous les joueurs peuvent attendre jusqu’à deux tours. Les cartes continueront d’être distribuées jusqu’à ce qu’il n’y ait qu’une seule rangée de cartes restantes, et la dernière rangée de cartes sera automatiquement attribuée aux joueurs qui n’ont pas sélectionné de carte. Après cela, chaque joueur entre dans la deuxième partie.

La deuxième partie du joueur est une partie multithread. Après qu’un joueur a choisi une certaine colonne de ses cartes, le joueur entre dans son propre thread. Premièrement, le joueur doit déterminer l’utilisation de toutes les cartes de cette colonne : comme un dessin carte ou En tant que carte de compétence. Si le joueur utilise une carte qui conserve une carte ou devient le joueur initial, il obtiendra une nouvelle valeur, qui sera retournée à l’objet de la classe principale. Après cela, le système déterminera si tous les pinceaux du joueur ont été placés sur la carte pour déterminer si le joueur peut mettre le pinceau sur la carte de compétence. Après cela, il continuera à déterminer si le joueur a un pinceau sur la carte pour que le joueur déplace le pinceau. Enfin, il sera déterminé si le joueur a les compétences nécessaires pour conserver la carte et s’il existe des cartes réservées et si les cartes réservées ont été utilisées, puis le joueur est autorisé à choisir s’il souhaite utiliser les cartes réservées.

Une fois les threads respectifs terminés, chaque joueur aura une valeur, et en appelant une fonction get au lieu d’utiliser la valeur de retour pour déterminer si le joueur deviendra le joueur initial, de sorte qu’il se limitera à choisir des cartes au tour suivant. .

Après chaque grand tour, les joueurs parcourront deux parties jusqu’à ce que le nombre de cartes à piocher pour un joueur atteigne 11 ou que toutes les cartes aient été distribuées.

Après cela, l’objet DCards de chaque joueur appellera la fonction de calcul du score pour calculer le score de chaque joueur.

À la fin, le joueur avec le score le plus élevé gagne.

Optimisation

En train d’établir une connexion avec la partie View, j’ai trouvé que dans le processus de vraies cartes, un nouvel objet JLabel est créé par la carte.Je pense que si l’objet JLabel est construit dans la classe NormalCard, chaque NormalCard a un indépendant Le JLabel n’a pas besoin de créer un nouveau contenu d’affichage lors de l’affichage de chaque carte, mais peut être obtenu directement en accédant à la fonction de NormalCard.

Pour déterminer si le joueur a choisi d’attendre dans le tour et si une certaine colonne a été sélectionnée, j’ai utilisé ArrayList pour ajouter et supprimer des objets, puis j’ai remplacé l’ArrayList d’origine au tour suivant.Je n’ai pas trouvé de moyen plus pratique de traiter avec elle, mais j’ai utilisé La méthode prendra beaucoup de mémoire.

Dans le jeu final, les joueurs doivent toujours respecter la logique normale du jeu et ne pas créer délibérément de situations pouvant causer des problèmes.Un traitement invisible est effectué, mais le système reconnaît toujours l’existence de ce type d’opération.Ce type de problème doit être capable d’ajouter la gestion des exceptions et d’utiliser try-catch dans la partie où le problème peut survenir.

Dans le thread de chaque joueur, la partie pinceau est conçue pour être déplaçable à tout moment, ce qui est plus logique. Si vous devez implémenter cette étape, vous devez utiliser le nombre de pinceaux mobiles du joueur comme ressource partagée. Dans le wait() état, une fois le mouvement du pinceau terminé, utilisez notifyAll().

Dans la dernière mise à jour du code, j’ai essayé de stocker toutes les variables dans une classe Data. Chaque classe doit construire ses objets dans cette classe, mais au final, le code ne sera pas rationalisé, mais beaucoup seront créés. get and set les fonctions.

Dans l’idée originale, nous avons l’intention d’afficher une page appartenant au joueur séparément après que chaque joueur entre dans le multi-thread, mais en raison de la conception basée sur la classe Swing, un panneau peut être exécuté en plusieurs Le format du fil.
